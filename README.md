# Furniture Assembly Bot

Telegram-бот для приёма заявок на сборку мебели и распределения их между мастерами.

## Структура проекта

```
furniture-bot/
├── bot/
│   ├── config.py        # чтение и валидация переменных окружения
│   ├── database.py      # SQLite-хранилище заявок (aiosqlite)
│   ├── keyboards.py      # инлайн-клавиатуры
│   ├── states.py         # FSM-состояния
│   ├── handlers/
│   │   ├── client.py     # диалог с клиентом: /start -> услуга -> район
│   │   └── master.py     # "Взять заказ" и ответ мастера клиенту
│   └── main.py           # точка входа: aiohttp + webhook
├── requirements.txt
├── .env.example
├── Procfile
└── README.md
```

## Как теперь устроена заявка (квиз + жизненный цикл)

Клиент проходит пошаговый квиз в боте: имя → телефон (кнопкой или текстом) →
адрес → этаж → лифт (да/нет) → тип мебели → описание → фото (можно
пропустить) → дата → время → комментарий → экран "всё верно?" перед
отправкой.

После подтверждения в чат мастеров уходит карточка вида:

```
📋 Заявка №154
👤 Иван
📞 +7 999 123-45-67
📍 Москва, ул. Ленина 15к2
🏢 Этаж: 8 | Лифт: ✔️ есть
🪑 Шкаф 🗄: Hoff, 4 двери
📅 24.07   🕒 14:00
📝 Закрепить к стене
```
+ кнопка **«🛠 Взять в работу»**.

Статусы заявки:

- **new** — свободна, видна всем мастерам с кнопкой "Взять в работу".
- **taken** — мастер взял в работу; кнопка в группе меняется на текст
  "🕐 В РАБОТЕ: @мастер"; мастеру в личку падает та же карточка с двумя
  кнопками: **"✅ Договорились"** и **"❌ Не получилось"**.
- **confirmed** — финал: мастер подтвердил, что созвонился и договорился с
  клиентом. Клиенту приходит уведомление, что мастер подтвердил заказ.
- Если мастер жмёт "❌ Не получилось" — статус возвращается в **new**,
  карточка в группе снова становится активной с кнопкой "Взять в работу",
  и любой другой мастер может забрать заявку.

Цена в боте никак не участвует — по вашей модели мастер сам обсуждает
стоимость с клиентом по телефону после того, как взял заказ в работу.
Бот отвечает только за учёт и передачу контактов/деталей.

## Что изменилось по сравнению с исходным вариантом

- **Заявки хранятся в SQLite, а не в `ORDERS_DB = {}`.** Раньше при перезапуске
  (деплой, падение процесса) все заявки терялись безвозвратно.
- **"Взять заказ" стало атомарной операцией.** Раньше два мастера, нажавшие
  кнопку одновременно, теоретически могли оба пройти проверку `is_taken` до
  того, как флаг выставится. Сейчас это один `UPDATE ... WHERE status='new'`,
  и выигрывает ровно один запрос.
- **Валидация конфига при старте.** Если забыли `BOT_TOKEN` или
  `MASTERS_CHAT_ID` — бот падает сразу с понятной ошибкой, а не в рантайме
  на первом сообщении.
- **Логирование** вместо тишины: старт вебхука, ошибки отправки сообщений,
  проблемы с мастером, который не открывал бота в личке.
- **Обработка ошибок Telegram API** — например, если клиент заблокировал
  бота или мастер ни разу не писал боту (Telegram не даёт написать первым).
- **Секрет вебхука** (`WEBHOOK_SECRET`) — защита от того, что кто-то
  подделает POST-запрос на ваш `/webhook` в обход Telegram.
- **Health-check `GET /`** — многие хостинги (Render и т.п.) пингуют
  корень, чтобы понять, что сервис жив.
- **Код разбит на модули** вместо одного файла — конфиг, БД, клавиатуры,
  хендлеры клиента и мастера. Так проще добавлять новые услуги, тестировать
  и не терять код при росте проекта.
- **Валидация ввода района** (длина текста) — раньше в базу улетало что
  угодно, включая случайно отправленный стикер/фото.

## Запуск локально

```bash
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env
# заполните .env своими значениями, затем экспортируйте переменные, например:
export $(cat .env | xargs)

python -m bot.main
```

Локально для вебхука понадобится публичный URL (например, через ngrok:
`ngrok http 10000`), либо на время разработки можно временно переписать
`main.py` на `dp.start_polling(bot)` — но в проде используется вебхук.

## Деплой (Render / аналоги)

1. Залейте проект в git-репозиторий.
2. Создайте Web Service, укажите команду `python -m bot.main`
   (уже прописана в `Procfile`).
3. Задайте переменные окружения из `.env.example` в настройках сервиса.
4. `WEBHOOK_URL` должен указывать на публичный адрес сервиса + `WEBHOOK_PATH`,
   например `https://mybot.onrender.com/webhook`.

## Дальнейшие идеи для развития

- Команда `/mystats` или `/orders` для мастеров — список их текущих заказов.
- Ограничение: один мастер не может держать больше N незакрытых заказов.
- Таймаут: если заказ не взяли за N минут — напомнить в чат мастеров.
- Отдельная таблица `masters` с рейтингом/специализацией.
